# 🏛 Architecture Decision Records (ADR)

---

# PURPOSE

This folder contains all major architectural decisions made during the evolution of the Metadata-Driven Plugin Runtime.

An ADR explains:

- What decision was made
- Why it was made
- Alternatives considered
- Consequences

Architecture should evolve through ADRs instead of undocumented discussions.

---

# ADR STATUS

Each ADR must have one status.

- Proposed
- Accepted
- Superseded
- Deprecated

---

# FILE NAMING

Use incremental numbering.

Example:

ADR-0001-zero-trust.md

ADR-0002-capability-system.md

ADR-0003-signed-manifest.md

ADR-0004-assemblyloadcontext.md

---

# TEMPLATE

Every ADR should contain:

- Context
- Decision
- Alternatives
- Consequences
- References

---

# PRINCIPLE

Architecture decisions are immutable historical records.

Never rewrite history.

Create a new ADR instead.
